# Jarvis Working Memory

Last updated: 2026-06-15 18:13 CST

This is Codex's local project notepad for Jarvis. It is intentionally informal:
use it to preserve short-term goals, product principles, test prompts, and
implementation context that should survive interruptions.

## Current Focus

- Keep Jarvis moving toward a full speech-in/action-out assistant loop.
- Maintain a real product surface: fast responses, visible text, reliable audio,
  and a menu-bar Shut Up control that is always reachable when Jarvis can speak.
- Treat every user-facing workaround as temporary unless it is good enough for
  the final Jarvis experience.
- Always keep the current task stack explicit. If work goes down nested layers,
  write down the top-level goal, the current branch, the immediate subtask, the
  proof needed to finish it, and the parent task to return to afterward.

## Goal Stack Discipline

Jarvis work can easily become nested: product area, subsystem, model/tool, bug,
then a tiny implementation decision. Do not lose the thread.

Before or during substantial work, keep this shape in mind:

- Top-level goal: what Leo ultimately wants Jarvis to become or do.
- Current branch: UI, Logic, Models, Skills, Voice, Browser, Music, Calendar,
  Codex, Memory, Safety, or another concrete lane.
- Immediate subtask: the specific thing being fixed or built right now.
- Why this matters: how the subtask moves the top-level goal forward.
- Done proof: the exact evidence needed, such as a test, live screenshot,
  endpoint result, voice-loop transcript, app behavior, or verifier report.
- Return point: the parent task to resume after this subtask is done.

If the work enters a rabbit hole, pause briefly and restate the stack. Do not
declare success just because a nested subtask passed; climb back up and continue
from the parent objective.

## Notes On Expected Behavior

- Teams task: use browser/bookmarks and logged-in Chrome when needed; do not
  pretend WebKit inherited Chrome cookies.
- Music task: LocalOS should control/play the Music app/player. Jarvis should
  only claim playback after confirmation. `accepted` is only an internal status;
  after the evidence window closes without actual audio, Jarvis should say
  LocalOS did not start the audio. If LocalOS is not connected, Jarvis should
  attempt to open/reconnect the music player automatically before asking Leo to
  do anything.
- RAM task: read real system memory state, ideally through a reliable native or
  terminal-backed tool.
- Codex task: choose the correct Codex chat, defaulting weird/general prompts to
  the Default chat, and mark prompts as Jarvis-generated.
- Calendar task: use local Calendar cache when permissions allow it; explain
  permission problems clearly.
- Model test task: avoid heavy local runs on Leo's 16 GB MacBook Pro when a
  cloud or MacBook Air route is safer.
- Ms. Sharpay task: infer contact aliases from local sender metadata and store
  useful contact mappings without leaking private content.
- Magic Keyboard task: `commerce.price_convert` now uses Apple's official U.S.
  Magic Keyboard (USB-C) product page plus a live USD-CNY public exchange-rate
  source. Keep replies short and spoken-friendly; source URLs stay in JSON.

## Product Principles From Leo

- No fake keyword-only skill demos. Let the model choose tools from clear tool
  definitions and keep tool calls hidden from spoken output.
- If Jarvis starts doing work, it should visibly and audibly acknowledge the
  task naturally, then return with the result.
- Spoken output must be clean English unless a specific non-English phrase is
  necessary.
- Never speak internal model/tool/debug details to Leo.
- If Jarvis is speaking, a visible menu-bar head with Shut Up must be available.
- Use tests for every bug Leo catches, especially speech mismatch, music false
  positives, and UI/menu-bar regressions.
